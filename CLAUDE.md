# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

Sigma Touch is an Android app that ports PC games to touch devices, built on the shared OpenTouch infrastructure. It was scaffolded from Psi Touch and currently ships two engines: **UE1** (Unreal Engine 1 / Unreal, module `:Unreal`) — verified running with full touch controls — and **UT99** (Unreal Tournament, engine v400, module `:UT99`) — a later UE1 generation, at Phase 2 (builds and boots on-device; menu is mouse-navigable, resolution + UI scale work; gameplay input still stubbed — see "UT99 (Unreal Tournament) engine notes" below). See [`docs/porting/phase1.md`](docs/porting/phase1.md). The app itself is a thin host: nearly all UI, controls, storage, and SDL plumbing live in the `AndroidCore_OpenTouch` library submodule.

- **App module namespace:** `com.opentouchgaming.sigmatouch`
- **Min SDK:** 19 (Android 4.4) · **Target/Compile SDK:** 35
- **Languages:** Java + Kotlin (app), C/C++ (native libs)
- **NDK:** 24.0.8215888 · ABIs: `armeabi-v7a`, `arm64-v8a`

## Repository Layout

This Git repo is a multi-submodule super-project. The Gradle project root is `SigmaTouch/` (not the repo root).

- `SigmaTouch/` — Gradle root (`settings.gradle`, `build.gradle`, `gradlew`). Run all Gradle commands from here.
- `SigmaTouch/sigmatouch/` — the app module (`:sigmatouch`).
- `AndroidCore_OpenTouch/` — submodule providing the `:androidcore` and `:changeloglib` library modules. **Most shared behavior lives here; see its own `CLAUDE.md`.**
- `SAFFAL/` — submodule providing `:saffal` (scoped-storage file abstraction).
- `SigmaTouch/sigmatouch/src/main/jni/` — native sources, mostly submodules (`SDL2_OpenTouch`, `Clibs_OpenTouch`, `MobileTouchControls`, `gl4es`, `AudioLibs_OpenTouch`).
- `SigmaTouch/sigmatouch/src/main/jni/Games/` — each game engine wrapped as its own Gradle library module, registered in `settings.gradle`. Currently: `Games/Unreal/` (`:Unreal`, engine source at `Games/Unreal/src/main/jni/UE1`) and `Games/UT99/` (`:UT99`, engine source at `Games/UT99/src/main/jni/ut99dc`).

After cloning, initialize submodules: `git submodule update --init --recursive`.

## Build Commands

Run from `SigmaTouch/`:

```bash
./gradlew assembleDebug        # Build debug APK
./gradlew installDebug         # Build + install on connected device
./gradlew clean
./gradlew :Unreal:assembleDebug -PabiOverride=arm64-v8a   # One engine, one ABI (fast iteration)
```

Command-line builds need a JDK ≤ 21 — the system default may be newer and breaks Gradle 8.13 ("Unsupported class file major version"). Use Android Studio's JBR: `JAVA_HOME="/Applications/Android Studio.app/Contents/jbr/Contents/Home" ./gradlew ...`

### Native build model

`src/main/jni/Android.mk` builds SDL2, touchcontrols, gl4es, audio libs, SAFFAL, etc. into `src/main/libs/<abi>/*.so` via `ndk-build`. The app module's own `externalNativeBuild` points at `Android_dummy.mk` (empty) on purpose, so Gradle does not rebuild these automatically — rebuild them with `ndk-build` against `Android.mk` when the underlying submodules change. Engine modules (`:Unreal`) build through CMake and link against these prebuilt `.so`s.

## Architecture

### Host ↔ Core split

The app provides only app-specific configuration and engine wiring; the heavy lifting is in `:androidcore`.

- **`EntryActivity`** (`:sigmatouch`) — launcher activity. Its static initializer registers `AppInfo.gameEngines` (currently just UNREAL), tutorials, scoped-storage tutorial, storage examples, and user-file entries. Calls `AppInfo.setAppInfo(... Apps.SIGMA_TOUCH ...)` in `onCreate`. **To add an engine, add a `GameEngine` entry to the `gameEngines` array here.**
- **`SigmaFragment extends MainFragment`** (`:androidcore`) — owns per-engine `GameLauncherInterface` instances (`UE1Launcher`) and the `launchGame()` flow. `setLauncher()` switches the active launcher based on `AppInfo.currentEngine.engine`.

### Adding an engine

Each engine needs three pieces wired together (see Psi Touch for concrete examples):

1. A `GameLauncherInterface` impl (e.g. `TFELauncher.kt`) registered in `SigmaFragment`.
2. An `EngineOptionsInterface` impl under `engineoptions/` — settings dialog + `RunInfo`, referenced from the `GameEngine` entry in `EntryActivity`.
3. A native engine module under `jni/Games/` registered in `SigmaTouch/settings.gradle` and depended on in `sigmatouch/build.gradle`.

`SIGMA_TOUCH` is registered in `AppInfo.Apps` and shares Psi's gamepad definition in `GamepadDefinitions` (both in `:androidcore`).

### UE1 (Unreal) engine notes

The `:Unreal` module wraps **UE1** — Unreal Engine 1 v200 source with community fixes (upstream `fgsfdsfgs/UE1`, branch `64bit_ai_fix`; see `AGENTS.md` in the engine root for the 64-bit port context). Currently a plain checkout, **not yet registered in `.gitmodules`** — fork to `emileb/UE1` and `git submodule add` is still pending. Status: **verified running on-device** — boots to the menu, plays, real audio, gameplay input, sane render resolution and text/HUD scale all confirmed working.

Where it deviates from the TFE/OpenJK template:

- **Single static `libunreal.so`.** UE1 is normally modular (`Core.so`, `Engine.so`, `NSDLDrv.so`, ... + launcher exe, loaded by name at runtime). On Android `Source/CMakeLists.txt` forces `BUILD_STATIC=ON` (plus `BUILD_EDITOR=OFF`, `BUILD_NOPENALDRV=ON`) under `if(ANDROID)`, so every package is a static lib whole-archive-linked into one shared `Unreal` target — same proven path as the engine's Vita/PSP ports (static module registration via `UNREAL_STATIC` / `UnStaticExports`). The target gets `PREFIX "lib"` + `OUTPUT_NAME "unreal"` because UE1 clears the `lib` prefix globally.
- **Do not link `NSDLDrv` directly into the `Unreal` target on Android** — it is already inside the `--whole-archive` group; naming it twice makes lld error with duplicate symbols (guarded in `Source/Unreal/CMakeLists.txt`).
- **`CMAKE_SOURCE_DIR` fixes.** Built standalone, UE1's CMake sits at the top, so it used `CMAKE_SOURCE_DIR` for `cmake/` and `Thirdparty/` paths. Under our wrapper (`add_subdirectory`) that resolves to the wrapper dir. Replaced with `CMAKE_CURRENT_SOURCE_DIR` (pre-`project()`) / `UE1_SOURCE_DIR` (auto-defined by `project(UE1)`) — behaviour-identical for desktop builds. Expect the same issue in other engines whose CMake assumes it is top-level.
- **Renderer:** the engine's own `NOpenGLESDrv` (OpenGL ES 2.0). Its `glad_es` loader resolves GL functions through `SDL_GL_GetProcAddress`, so no `GLESv2` link is needed. `RunInfo.glesVersion = 2`.
- **Shader precision: everything is `highp`.** `NOpenGLESDrv/{Vertex,Fragment}Shader.glsl.inc` originally declared `aPosition`/texcoords/etc as `mediump`/`lowp`. Desktop GL (and most Android GPUs) ignores GLES precision qualifiers and just uses full float regardless, so this compiled and looked fine everywhere *except* real precision-enforcing hardware (PowerVR is the classic offender) — there it's as low as ~10 bits of mantissa, causing visible vertex jitter/swimming and texture-UV instability as the camera moves away from the origin. Fixed by declaring every attribute/varying/uniform/local `highp` (plus `precision highp float;`/`precision highp sampler2D;` defaults in the fragment shader). If a future engine's GLES shaders show the same symptom on real hardware but look fine on desktop/emulator, check precision qualifiers first.
- **UI/HUD scale is decoupled from render resolution — added, not built in.** `Canvas.ClipX`/`ClipY` (all UnrealScript HUD/menu/text layout) is set 1:1 from `FSceneNode.X`/`Y`, which is *also* what sizes the GL viewport and the 3D frustum (`Render/Src/UnRender.cpp`'s `CreateMasterFrame`, `NOpenGLESDrv.cpp`'s `SetSceneNode`/`DrawTile`) — there's no separate "UI scale" concept, and the actual menu/HUD drawing code lives in the *compiled* `Engine.u`/`Unreal.u` game data, so it can't be patched to scale itself. At a high selected resolution, its fixed-pixel icons/fonts become proportionally tiny. Fixed at the engine level (`#ifdef __ANDROID__`): `CreateMasterFrame` now reports a small fixed logical canvas (480px tall, width matching the real aspect ratio) instead of the true pixel size, while `SetSceneNode`'s `glViewport()` call explicitly uses the real `Viewport->SizeX`/`SizeY` for the root frame (child/mirror frames untouched) — since NDC space is resolution-independent, 2D drawing computed in the small logical space automatically stretches to fill the real viewport, and 3D perspective is unaffected (only the aspect ratio, preserved by construction, and FOV matter there). **General lesson**: any legacy engine whose 2D UI is drawn in fixed device pixels (not a scale-invariant unit) will have this problem once you let the player pick a modern resolution — look for wherever the render target size feeds both the GL viewport *and* the UI/canvas coordinate space, and split them.
- **Resolution is player-selectable via `-ResX=`/`-ResY=`, forced independent of anything in the .ini.** UE1 has no native resolution command-line flag — `ViewportX`/`ViewportY` (`Engine/Inc/UnCamera.h`) are plain `CPF_Config` ints, ini-only. Added `Parse(appCmdLine(), "ResX=", Client->ViewportX)` (and `ResY`) right after `Client->Init()` in `UGameEngine::Init()` (`Engine/Src/UnGame.cpp`) — a small, non-Android-gated, generically useful addition. `EngineOptionsUnreal.kt` wires this to the shared `ResolutionOptionsWidget` (same widget/pattern `EngineOptionsOpenJKBase` uses), building `info.args = " -ResX=${res.w} -ResY=${res.h} "`. Presets like "Screen (100%)" pass literal `$W`/`$H`-style placeholder tokens as `res.w`/`res.h` — safe, because `SDLOpenTouch.java`'s `ReplaceDisplaySize()` substitutes those in the `args` string with real pixel values before the native process starts.
- **Sound:** `NOpenALDrv` is built and forced as the `AudioDevice` regardless of `Default.ini`/`User.ini` — see `UEngine::Init()` in `Engine/Src/UnEngine.cpp`: a plain (non-`"ini:"`-prefixed) class path bypasses the config lookup entirely (`GObj.LoadClass` only does the `"ini:"` indirection for strings with that prefix). It needs `openal-soft` (wrapper CMake imports the `:openal` output) plus `libxmp` (tracker/MOD music) — both real Gradle+CMake modules under `AudioLibs_OpenTouch/` now (`openal-soft/`, `libxmp/`, the latter built with `-DBUILD_LITE=ON -DBUILD_SHARED=ON -DBUILD_STATIC=OFF` producing `libxmp-lite.so`), mirroring each other's pattern exactly (wrapper `CMakeLists.txt` importing the module's `merged_native_libs` output, `IMPORTED SHARED` in the engine's own wrapper CMake, explicit link on the final `Unreal` target since whole-archived static libs only reference them by raw path). Needed a `pickFirst "**/libxmp-lite.so"` (and already had one for `libopenal-soft.so`) in `sigmatouch/build.gradle`'s `packagingOptions.jniLibs` — any `.so` that's both `implementation(project(":x"))`'d directly by `:sigmatouch` *and* referenced as `IMPORTED SHARED` inside an engine module's CMake gets auto-copied into that engine module's own jniLibs too, so it collides at final merge time otherwise.
- **Touch glue:** UE1 has its own `mobile/touch_interface_unreal.cpp` + `mobile/touch_interface.h` (not the shared `Clibs_OpenTouch/alpha/touch_interface.cpp` Phase 1 started with — that file also carries other Alpha Touch engines' `PERFECT_DARK`/`ROTTEXPR`-specific layout branches, which don't apply here and were dropped when copying it over). The wrapper CMake puts `mobile/` on the include path (not `${CLIBS_DIR}/alpha`) so the shared `Clibs_OpenTouch/android_jni_inc.cpp`/`touch_interface_base.cpp` resolve the same `TouchInterface`/`touch_interface.h` this engine's own copy uses (the ODR-mismatch trap the general porting guide above warns about). `PortableInit` calls the engine's `main(argc, argv)`. Do **not** define `isPlayerRunning` / `MouseButton` / `MouseMove` / `MouseMoveAbsolute` in `game_interface.cpp` — `android_jni_inc.cpp` defines those itself.
- **Gameplay actions bypass UE1's key-bindings, not just its keys.** `mobile/game_interface.cpp` implements movement/look/fire/jump/weapons for real (not stubs). Menu navigation, the on-screen-keyboard passthrough and the generic custom buttons are still synthesized as real SDL key/mouse events (`SDL_SendKeyboardKey` / `MouseButton`) — correct there because Unreal's menu/console (`Console.uc`) reads raw `EInputKey` codes directly (`UEngine::InputEvent` asks the console before the rebindable path even runs) and the custom buttons are meant to be rebindable like any other key. Everything else (movement, turning, fire, jump, weapon switching, ...) calls the engine's real action directly — `UViewport::Exec()` with the exact alias name or raw `Axis`/`Button` command a bound key would reach (e.g. `"Fire"`, `"MoveForward"`, `"Axis aBaseY Speed=+300.0"`) — instead of depending on `Bindings[key]`, so rebinding controls in-game can't break these. Continuous movement/turn axes additionally need UE1's own per-tick "held key" pump (`KeyDownTable` + `UInput::ReadInput()`, `Engine/Src/UnIn.cpp`) to accumulate smoothly, which only triggers via `UInput::Process()` against a real `EInputKey` — so those go through a handful of reserved, otherwise-unreachable key slots (`IK_Unknown88`..`IK_Unknown8F`; no keyboard/mouse can generate these, see `NSDLViewport::InitKeyMap`) bound once to the real alias, then pressed/released like any other key. **Threading**: UE1 runs single-threaded on its own thread (`PortableInit` → `main()` → `MainLoop()`, which never returns), separate from the Android UI/touch thread that calls `Portable*`. So `Portable*` only ever sets plain `volatile` flags/floats or pushes onto a small ring buffer — never touches engine state directly. `UE1_TickPortableActions()` (defined in `game_interface.cpp`, called once per tick from `MainLoop()` in `Unreal/Src/SDLLaunch.cpp`) is what actually calls `Process()`/`Exec()`, and only ever runs on the engine's own thread. `NSDLViewport.cpp` exposes `UE1_GetViewport()` / `UE1_IsMenuActive()` (Android-only) for this — the engine has no `GEngine`-style global, so a `static UNSDLViewport*` set in the viewport's constructor/`Destroy()` stands in for one.
- **Per-app Clibs wrapper:** `Clibs_OpenTouch/sigma/android_jni.cpp` (one-line include of `android_jni_inc.cpp`) was added to the Clibs submodule for Sigma, mirroring `psi/`.
- **App wiring:** `UNREAL` added to the `Engine` enum in `:androidcore`'s `GameEngine.java`. `EntryActivity` entry: `loadLibs {"touchcontrols", "z", "unreal"}`, `EngineOptionsUnreal`. `UE1Launcher` uses folder **`UE1`** (run dir `/OpenTouch/Sigma/UE1`), detects an install by `System/Engine.u` + `Maps/Entry.unr` (present in retail and the v205 demo). `EngineOptionsUnreal` (`dialog_options_unreal.xml`) has the shared `ResolutionOptionsWidget` (see above) and a gamepad-config widget. `unreal_icon.png` is a generated placeholder — replace with real art.
- **Content compatibility:** this engine generation loads package versions ≈v61–67 (Unreal retail v200/205 demo). Unreal Gold / 225 / 226 content is v68 and **cannot run** (fails with a clean error) — see the engine's `AGENTS.md`.
- **Logging:** `Core`'s `FGlobalPlatform::WriteBinary` (`Core/Src/UnPlat.cpp`) mirrors every log line to `LOGI` and `LogWritter` (Android-only), so engine logs show up in `adb logcat` and in the app's shareable log file.
- **SAF (secondary/SD storage) needs absolute paths — relative + chdir only works on real filesystem storage.** `UGameEngine`'s default file lookup (`Engine/Config/Default.ini`'s `Paths[0..4]`/`SavePath`/`CachePath`, e.g. `Paths[1]=..\Maps\*.unr`) is CWD-relative — the launcher sets CWD to `.../UE1/System` via `game_path`, and `..\Maps\` walks back up to a sibling folder. That's fine on primary/real-filesystem storage (a real `chdir()` + real relative opens), but SAFFAL's libc interceptor (`SAFFAL/saffal/src/main/jni/Utils.cpp`, `getCanonicalPath()`/`isInSAF()`) decides "is this path SAF-backed" purely by checking whether the literal path *string* starts with its registered SAF root — it does **not** call `getcwd()`/`realpath()`, so a relative `"../Maps/Foo.unr"` can never match, and silently falls through to a real (failing) `fopen`. Fixed with a new `-GamePath=<abs>` argument, parsed lazily in `appFindPackageFile` (`Core/Src/UnPlat.cpp`): rewrites any `"../Xxx/..."`-shaped lookup to `"<GamePath>/Xxx/..."`, where `-GamePath=` is the UE1 folder itself (`System`'s parent, matching what `"../"` naturally resolves to) — done for the local `Temp` in the `Paths[]` search loop, and once, in place, for `GSys->SavePath`/`CachePath` (both real `char[96]` fields, so every other direct use of them elsewhere - save-game code in `UnGame.cpp`, download cache in `UnChan.cpp` - picks up the fix for free). **Not** applied to `GSys->Paths[]` itself — it's a `char[96][16]`, too small to hold an absolute path, hence the local-buffer rewrite instead. `UE1Launcher.kt.getArgs()` only passes `-GamePath=` when `subGame.rootPath == secondaryDirectory` (same condition `TFELauncher` uses for `--gamePath`) — primary storage keeps the original, already-working relative/chdir behaviour untouched. **General lesson**: a SAF/scoped-storage interceptor that matches by path prefix needs genuinely absolute, root-prefixed paths end to end — chdir + relative opens is a real-filesystem-only pattern, and any engine using it for its default asset search needs the same kind of rewrite before SAF storage will work, not just a "point it at the right folder" argument.
- **`Unreal.ini` isn't always writable in `System/`** (SAF-backed installs, or any read-only mount). Fixed by seeding a real, always-writable copy: `assets/Unreal.ini` (a known-good config) gets copied to `user_files/unreal/Unreal.ini` on first run (`UE1Launcher.checkForDownloads()`, via `Utils.copyAsset`), and `getArgs()` always passes `-INI=<absolute path to that copy>` (UE1 already supports `-INI=` natively — `Core/Src/UnPlat.cpp`'s `appInit()`). One nuance: UE1's stock `-INI=` handling *appends* the value to `appBaseDir()` (which is the literal string `"./"` on Android), so an absolute `-INI=` value naively appended collapses back into a CWD-relative path, not the real absolute one — patched `appInit()` to use an absolute `-INI=` value directly (skip the `appBaseDir()` prefix) instead of concatenating.

- **All user-generated files (saves, cache, screenshots) always redirect to `user_files/unreal/`, regardless of primary/SAF storage.** `GSys->SavePath`/`CachePath` are unconditionally rewritten (not just for the SAF case above) in `UnPlat.cpp`'s `AndroidFixupGSysPathsOnce()`, via a new `AndroidUserFilesDir()` helper that reads `getenv("USER_FILES")` (already set for every engine by `Clibs_OpenTouch/android_jni_inc.cpp` — same env var TFE's `paths-posix.cpp` reads) and appends `/unreal`; save-game code (`UnGame.cpp`) and download-cache code (`UnChan.cpp`) pick this up for free since they only ever go through `GSys->SavePath`/`CachePath`. Screenshots (`Engine/Src/UnCamera.cpp`'s `SHOT` console command) needed a separate fix — it built a bare relative `"Shot%04i.bmp"` filename with no path at all, writing straight into CWD (the game's own, possibly-read-only `System` folder) — patched (`__ANDROID__`-gated) to build the same `<USER_FILES>/unreal/Screenshots` path and `appMkdir()` it once before the exists-check loop.
- **SaveGame/LoadGame/HubPop build a `GSys->SavePath`-relative filename and hand it straight to `FURL`** (`UGameEngine::Browse()`'s `?load=`/`?pop=` handling and `SaveGame()`, all in `Engine/Src/UnGame.cpp`) — once `SavePath` became an absolute Android path (see above), this broke two different ways: (1) those call sites hardcoded a literal `\\` separator instead of `PATH_SEPARATOR`/`PATH()`, so the joined path had one raw backslash character where every other separator was `/`, which no POSIX `fopen` recognises as a directory separator (fixed by matching the `PATH_SEPARATOR`/`PATH(GSys->SavePath)` style already used a few lines below in the same functions — `UnChan.cpp`'s download-cache path had the identical bug); (2) even with separators fixed, `FURL`'s constructor (`Engine/Src/UnURL.cpp`) silently **resets the whole URL to the class default** (`Map` becomes the config's default map, e.g. `Index.unr`, instead of the real save file) for any string it can't parse as a URL - and a POSIX absolute path trips this twice: a single leading `/` reads as the start of a malformed `//host` prefix (Windows absolute paths dodge this via an existing, unrelated `C:` bypass earlier in the same `if`), and even past that, its own "trailing slash = portal name" logic (which only ever checks for `/`, so a backslash-separated Windows path was always naturally immune) chops a real multi-directory Android path into garbage on the first `/` it finds. Fixed by adding a `IsPosixAbsolutePath` check (`URL[0]=='/' && URL[1]!='/'`) that (a) joins the Windows-drive-letter bypass so absolute Android paths skip the host/protocol parsing block entirely, and (b) disables the portal-slash split for the same paths. **General lesson**: a legacy engine's URL/path parser written for Windows may have absolute-path special cases that only recognise the drive-letter form (`C:\`) and use the *absence* of forward slashes as an accidental correctness assumption — a POSIX absolute path breaks both assumptions at once, and needs its own explicit bypass alongside the existing Windows one, not just a separator swap.
- **Config writes only reach disk at graceful engine shutdown — which Android routinely skips.** `FConfigFile::SetString()` (`Core/Src/UnConfig.cpp`) only marks the in-memory config `Dirty`; the actual file write happens in `FConfigFile::Write()`, called solely from `FConfigCache::Exit()` at normal engine shutdown (player picks "Quit", `MainLoop()` returns, `appExit()` runs). An Android process routinely dies without ever reaching that path (task swipe, OS memory pressure, just closing the app), silently discarding any config change made that session — including UnrealScript's `globalconfig` save-slot names (`UnrealSlotMenu.SlotNames`, written by `UnrealSaveMenu`'s explicit `SaveConfig()`): a save made and then loaded within the *same* run worked fine (the in-memory config was still correct), but the Load Game menu came up completely empty after actually restarting the app, since the slot-name list was never flushed to `Unreal.ini`. Fixed by making `FConfigFile::SetString()` call `Write()` immediately after marking dirty, `__ANDROID__`-gated. **General lesson**: any engine whose config/save system defers writing to disk until a graceful-shutdown hook needs that write moved to happen immediately (or on an app-lifecycle pause/stop callback) on Android — the OS can and does kill the process without ever running that hook.
- **Editor-less build breaks stock maps that carry baked-in editor artifacts.** The Editor package isn't compiled on Android (`BUILD_EDITOR OFF`), so its native classes (`TransBuffer`, `MyLevel`) don't exist. Some stock maps were saved in UnrealEd with an editor `TransBuffer`/`MyLevel` object baked into the package (a cooking leftover — not all maps have it). In-game those exports are `RF_LoadForEdit` and skipped ([UnLinker.h:404](SigmaTouch/sigmatouch/src/main/jni/Games/Unreal/src/main/jni/UE1/Source/Core/Src/UnLinker.h) + the `CreateExport` gate), so they're never actually constructed — but `UGameEngine::LoadMap`'s strict `LOAD_Verify` pass (`Engine/Src/UnGame.cpp`) runs `VerifyImport` on *every* import regardless of `RF_LoadForEdit`, can't resolve `Editor.TransBuffer` without the Editor module, throws `FailedImport`, and the map load is caught as **"Failed To Load Map"** — the level transition silently aborts (game-breaking, only for the affected maps; symptom in logcat is `Failed to load 'Editor': Can't find Class Editor.TransBuffer in file '…/Editor.u'`). On PC this never fires because `Editor.dll`/`Editor.u` satisfy the import even in-game. Fixed (`__ANDROID__`-gated, in `ULinkerLoad::VerifyImport`, `Core/Src/UnLinker.h`): when `!GIsEditor` and the failing import belongs to the **Editor** package (detected by walking the import's package chain to the top and matching `ObjectName=="Editor"`), warn and leave it unresolved instead of throwing — safe because the only exports that reference it are the editor-only ones already skipped. Non-Editor missing imports still fail loudly. **General lesson**: an editor-less console/mobile build of an engine whose map format can embed editor objects needs its loader's *eager* import-verification to tolerate unresolved editor-package imports — the run-time object gate (`RF_LoadForEdit`) alone isn't enough, because verification runs before (and independently of) that gate. **Not yet applied to `:UT99`** (same editor-less setup, but its v400 loader is a separate file) — port the same guard there only if a UT stock map hits the identical failure.

### UT99 (Unreal Tournament) engine notes

The `:UT99` module wraps **Unreal Tournament** — UE1 engine v400, package file version **68** (upstream is a Dreamcast port, `ut99dc`; the checkout still has its DC/KOS drivers). It is a *later* UE1 generation than `:Unreal` (v200, versions 61–67), and the two content generations are mutually incompatible — UT99's v68 packages **cannot** load on `:Unreal` and vice-versa — so they are deliberately separate modules/engines that coexist (`UNREAL` and `UNREAL_TOURNAMENT` in `GameEngine.Engine`). Run dir `/OpenTouch/Sigma/UT99`, single static `libut99.so`, options class `EngineOptionsUT99`, launcher `UT99Launcher` (detects an install by `System/Botpack.u` + `System/Engine.u`).

**Status: Phase 1 complete; Phase 2 well advanced — boots, menu is mouse-navigable, resolution + UI scale work on-device, audio (OpenAL) ported, ini/SAF conveniences ported, user-file (save/cache/screenshot) redirection ported, and gameplay input (movement/look/fire/jump/weapons) implemented.** The whole engine compiles under the NDK and whole-archive-links into one `libut99.so` with `-Wl,--no-undefined` passing (arm64 built; armeabi-v7a should follow but is untested). **Audio (OpenAL) is ported** — see "Audio (OpenAL)" below. **Gameplay input is ported** — see "Gameplay input" below (compiles/links; on-device verification pending). Not yet a git submodule — a plain checkout (fork to `emileb/ut99dc` + `git submodule add` pending), same as `:Unreal` started.

Phase 2 so far (all `#ifdef __ANDROID__`, verified on-device):

- **Menu detection uses `bShowWindowsMouse`, not `bShowMenu`.** `UT99_IsMenuActive()` / `UT99_GetViewport()` (NSDLDrv/Src/NSDLViewport.cpp, backed by a static viewport ptr set in the ctor / cleared in `Destroy`) drive `PortableGetScreenMode()`. UT's **UWindow** GUI menu sets `bShowWindowsMouse` (a UPlayer flag - the Windows-style cursor); `bShowMenu` is only the classic Unreal-1 `Menu.uc` flag. Check both, exactly like the engine's own capture logic (`UGameEngine::Tick`, UnGame.cpp).
- **Menu mouse is relative-drag, and needs SDL capture (relative mode) ON.** The UWindow cursor is driven by the *delta* path — relative motion → `MouseDelta()` + `IK_MouseX/Y` input events → the console script moves the cursor — **not** the absolute `MousePosition()` path (that updates `WindowsMouseX/Y` coordinates the UT console never uses for cursor movement; verified on-device with per-event logging after several wrong fixes assumed the opposite). `NSDLViewport::TickInput` only takes the delta path while `SDL_GetRelativeMouseMode()` is on, but `UGameEngine::Tick`'s pause logic — which *does* run on Android, since the SDL window isn't flagged fullscreen so `!IsFullscreen()` passes — releases capture whenever the menu opens, which froze the menu cursor until a tap re-captured via TickInput's click-to-capture branch ("tap once before the mouse moves"). Fixed at the top of `TickInput()` (`#ifdef __ANDROID__`): if relative mode is off, re-capture — on a touch screen there's no real cursor to give back, so capture is simply kept permanently on (toggles only on the engine's rare release, not every frame — SDL flushes queued mouse motion on every relative-mode toggle, so continuous forcing eats injected drags). `mobile/touch_interface_ut99.cpp`'s `mouseMove` does relative drag scaled to the fixed 480-tall logical UI canvas (not the real resolution — the cursor lives in logical space, so real-resolution scaling made it race) + tap-to-click; `tcMenuMain` has a full-screen invisible trackpad (arrow/enter buttons removed — menu is mouse-driven). `PortableKeyEvent`/`PortableBackButton` inject SDL keys via `SDL_SendKeyboardKey` (Escape = open/close menu). **Debugging lesson:** when injected input misbehaves, log the actual runtime state (`SDL_GetRelativeMouseMode()`, `bShowWindowsMouse`, per-event x/xrel via `__android_log_print` — `debugf` isn't visible this early) across a fail→workaround→work sequence before theorizing; and note which *direction* of state change accompanies "working", not just that a change fixes it.
- **`-ResX=`/`-ResY=`** parsed in `UNSDLViewport::OpenWindow` (NSDLViewport.cpp) as the highest-priority window size, above the ini's `[NSDLDrv.NSDLClient] WindowedViewportX/Y`. UT99 has no native resolution flag; `EngineOptionsUT99` passes them (+ the matching framebuffer-scaler size), same as `EngineOptionsUnreal`.
- **UI/HUD scale decoupled from render resolution** — same fix as UE1 (see the UE1 notes' "UI/HUD scale" item). `URender::CreateMasterFrame` (Render/Src/UnRender.cpp) reports a fixed logical canvas (480 tall, aspect preserved) for `Frame->X/Y`, so `Canvas.ClipX/ClipY` (all UnrealScript HUD/menu/text layout) is resolution-independent. **The UT-specific twist UE1 never hit:** UWindow renders 3D actor previews into a sub-rectangle of the canvas via `execDrawClippedActor` (the Player-Setup model/face preview) and `execDrawPortal` (UnCanvas.cpp) - both reuse/clone the *master* frame (`Parent==NULL`) with `X/Y/XB/YB` overwritten to a rect in **Canvas coords (now logical)**. So `UNOpenGLESRenderDevice::SetSceneNode`'s root-frame `glViewport` must **scale that logical rect back to real pixels** (uniform `S = SizeY/480`), not hardcode full-screen - hardcoding full-screen blows the preview up to fill the screen. The scale reproduces the exact real-pixel viewport the pre-logical-canvas code used, so it's correct for the full-screen HUD frame *and* the sub-rect previews. `UILogicalHeight` (480) must stay in sync between `CreateMasterFrame` and `SetSceneNode`. **General lesson:** once the 2D canvas is a logical space, every place that feeds Canvas-space coords into a GL viewport/scissor needs the logical→real scale, not just the main frame.
- **Font size:** the in-game Video option that "doubles the font size" is `[UMenu.UMenuRootWindow] GUIScale` (float; 1.0 normal, 2.0 doubled - the checkbox writes those two, but any float is accepted, though UWindow's bitmap fonts snap to fixed sizes). Lives in the compiled `UMenu.u` (not in this repo's source).

Two non-obvious things had to be fixed before it would boot (both classic "fine on desktop, broken on device"):

- **`RunInfo.args` must be non-null.** `EngineOptionsUT99.getRunInfo()` originally left `info.args` unset; `SigmaFragment` concatenates it into the launch command line, so a null Java `String` became the literal token `"null"`, which UE1's `UGameEngine::Init` (`Engine/Src/UnGame.cpp`) parsed as the startup map name → `Failed to enter null: Can't find file 'null'`. Fixed by setting `info.args` (to the `-ResX/-ResY` string, matching `EngineOptionsUnreal`; the leading `-` also makes `UnGame.cpp` fall back to the ini's default map). Any new `EngineOptions*` must set `args` to a non-null value.
- **GLES context profile (`NSDLDrv/Src/NSDLViewport.cpp`, `OpenWindow`).** When the render device name contains `GLES`, the code set `GLProfile = SDL_GL_CONTEXT_PROFILE_COMPATIBILITY` — desktop GL tolerates the wrong profile mask, but Android only has GLES, so `SDL_GL_CreateContext()` failed there (`Could not create GL context`). `__ANDROID__`-gated to request `SDL_GL_CONTEXT_PROFILE_ES` (a real ES2 context) on Android while desktop keeps the compatibility context that `NOpenGLESDrv::CreateShader`'s runtime `IsGLES` check depends on (it compiles `#version 120` for the non-ES desktop context). Requires the ini's `[Engine.Engine] GameRenderDevice`/`WindowedRenderDevice` to actually name `NOpenGLESDrv.NOpenGLESRenderDevice` — that string is what selects the ES path.

Because `ut99dc` already carries the SDL2 + GLES2 desktop drivers (`NSDLDrv`/`NOpenGLESDrv`, themselves transplanted from UE1) and an existing `BUILD_STATIC`/`UNREAL_STATIC` path (for Dreamcast), the Android port closely mirrors `:Unreal`'s and needed far less engine surgery:

- **Single static `libut99.so`.** `Source/CMakeLists.txt`'s new `if(ANDROID)` block forces `BUILD_STATIC=ON` and disables `BUILD_EDITOR`/`BUILD_DCUTIL`/`BUILD_AICADRV`/`BUILD_PVRDRV`/`BUILD_NULLSOUNDDRV`; `Source/UnrealTournament/CMakeLists.txt` gained an Android branch (mirror of UE1's `Unreal` target) that builds the final target as a `SHARED` lib (`PREFIX "lib"`, `OUTPUT_NAME "ut99"`, `ENGINE_NAME="ut99"`), does **not** name `NSDLDrv` in the link (it's already inside the `--whole-archive` group — duplicate-symbol trap), and pulls in the OpenTouch glue.
- **`-fsigned-char -fno-short-enums`** are added in the Android block: ARM defaults `char` to *unsigned*, but Core's `UnGnuG.h` has a `static_assert("char must be signed")`. Same flags UE1 uses.
- **`CMAKE_SOURCE_DIR` fixes** (the top-level-assumption trap): `Source/CMakeLists.txt`'s `list(APPEND CMAKE_MODULE_PATH …/cmake)` → `CMAKE_CURRENT_SOURCE_DIR` (runs before `project()`, so `UT99_SOURCE_DIR` isn't defined yet), and `Source/NOpenGLESDrv/CMakeLists.txt`'s three `${CMAKE_SOURCE_DIR}/../Thirdparty` → `${UT99_SOURCE_DIR}/../Thirdparty`. Behaviour-identical for desktop.
- **No viewport changes needed to boot.** UE1's `NSDLViewport` `__ANDROID__` blocks are *only* the Phase-2 touch hooks (`GAndroidViewport`/`UE1_GetViewport`/`UE1_IsMenuActive`); GL-context creation goes through the ordinary desktop `SDL_CreateWindow`/`SDL_GL_CreateContext` path, which SDL's Android backend handles. UT99's viewport is the same code, so nothing there was touched for Phase 1.
- **Logging:** two hooks, because UT99 has several separate `FOutputDevice`s rather than UE1's single `FGlobalPlatform::WriteBinary` funnel. (1) `FOutputDeviceFile::Serialize` (`Core/Src/FOutputDeviceFile.cpp`) mirrors every `debugf`/`GLog` line to `LOGI` (tag `UT99`) and `LogWritter_Write` (`TCHAR` is `ANSICHAR` here, so `Data` goes straight to logcat). (2) **Critical errors** (`appErrorf`, e.g. `FailedBrowse`, and the full guard-history dump) go through `FOutputDeviceAnsiError`, which `printf`s to stdout/stderr — invisible on Android — so `mobile/game_interface.cpp`'s `PortableInit()` installs a stdout/stderr→logcat pump (`pipe`+`dup2`+reader thread, tag `UT99`) before calling `main()`. Without hook (2), a startup `appErrorf` leaves no trace in logcat at all.
- **Mobile glue:** `mobile/game_interface.cpp` — `PortableInit()` starts the engine (calls `main()`; the `main`→`SDL_main` remap is consistent because both it and `Launch.cpp` include `SDL2/SDL.h`) and installs the stdout/stderr→logcat pump. Gameplay input is fully implemented — see "Gameplay input" below. `mobile/touch_interface_ut99.cpp`/`touch_interface.h` were copied from UE1's. The Clibs per-app wrapper `Clibs_OpenTouch/sigma/android_jni.cpp` is reused as-is.

- **Gameplay input (movement/look/fire/jump/weapons), Phase 2 — ported near-verbatim from the sibling UE1 module's `game_interface.cpp`** (compiles/links; on-device verification pending). UT99 is a later UE1 generation with the *same* `UInput`/`UViewport` input architecture, so the whole mechanism carried over 1:1; UT99's default aliases/axes even match Unreal's (`Fire`/`AltFire`/`Duck`/`Walking`/`Strafe`, `aBaseY`/`aBaseX`/`aStrafe` at the same speeds — confirmed against `assets/User.ini`'s `Aliases[]`). How it works: **menu nav / on-screen keyboard / custom buttons / mouse-look** are synthesized as real SDL key/mouse events (`SDL_SendKeyboardKey`/`MouseMove`/`MouseButton`) — correct and rebind-proof because UT's menu/console reads raw `EInputKey` codes directly and the custom buttons are meant to be rebindable. **Movement/turn/fire/jump/weapons** instead call the engine's real action (`UViewport::Exec("Fire")`, raw `Axis` commands) so player rebinds can't break them. **Continuous movement/turn/strafe** need UE1's per-tick "held key" pump, so they go through reserved, otherwise-unreachable key slots (`IK_Unknown88`..`8A`, values 136-138 — same as v200; no real device generates them) bound once to the raw axis command, then driven each tick via `UInput::Process(*GLog, key, IST_Axis, magnitude*100*dt)`. **Threading:** the engine runs single-threaded on its own thread; `Portable*` (touch thread) only sets `volatile` flags/floats or pushes a ring buffer; all `Process()`/`Exec()` calls happen in `UT99_TickPortableActions()` (game_interface.cpp), called once per tick from `MainLoop()` (`UnrealTournament/Src/Launch.cpp`, `#ifdef __ANDROID__`, just before `Engine->Tick`) on the engine thread. **v400 deltas vs the v200 driver:** `UViewport::Exec`/`UInput::Process` take `FOutputDevice&` (pass `*GLog`) not `*`; otherwise identical. **Include-order trap (again):** game_interface.cpp pulls in `<pthread.h>` (for the logger) — its transitive `<time.h>` must precede `Engine.h` or Core's `clock()` macro mangles `clock_t clock(void)`, so all C system headers are included before `Engine.h`.
- **Audio (OpenAL), Phase 2 — ported from `:Unreal`'s `NOpenALDrv`** (compiles + whole-archive-links into `libut99.so`; on-device verification pending). UT99 ships no usable audio backend of its own: its generic `Audio/` mixer (`UGenericAudioSubsystem`) only has an OSS `/dev/dsp` output (`AudioCoreLinux.cpp`, `#include <linux/soundcard.h>`) that doesn't exist on Android. So the UE1 module's OpenAL driver was ported wholesale into `Source/NOpenALDrv/` (openal-soft + libxmp, same Gradle+CMake wiring — wrapper `Games/UT99/CMakeLists.txt` imports the `:openal`/`:libxmp` module `.so` and creates `openal-soft`/`xmp-lite` IMPORTED targets; `Source/CMakeLists.txt` sets `BUILD_NOPENALDRV ON` on Android and adds it to `INSTALL_TARGETS` so it's whole-archived; the final `UnrealTournament` target links `openal-soft xmp-lite` explicitly; `build.gradle` `dependsOn`/`implementation`s `:openal` + `:libxmp`; `pickFirst` for both `.so` already in `sigmatouch/build.gradle`). Forced as the audio device on Android in `UEngine::InitAudio` (`Engine/Src/UnEngine.cpp`, `#ifdef __ANDROID__`): a plain (non-`"ini:"`) class path `NOpenALDrv.NOpenALAudioSubsystem` bypasses the config indirection and the `UseSound` check — a retail ini names Windows drivers and usually has `UseSound=False`. **v400 API deltas vs the v200 UE1 driver** (all handled in the ported copy): (1) property registration is `StaticConstructor()`/`GetClass()` (v200 used `InternalClassInitializer`); (2) `Exec( const TCHAR*, FOutputDevice& )` signature; (3) three extra `UAudioSubsystem` pure virtuals to implement — `GetViewport`/`RenderAudioGeometry`/`PostRender` (last two are no-ops here); (4) `USound::Data` is a `FSoundData : TLazyArray<BYTE>` — must `Load()` before `ReadWaveInfo`/upload and `Unload()` after (v200 was a plain `TArray<BYTE>` you `Empty()`'d); (5) `USound` lost its `Looping` flag, so looping is now decided per-slot (`SLOT_Ambient` loops) instead of from WAV loop points; (6) the fixed-array `DeviceName` config prop was dropped — v400 has no fixed-buffer `UStringProperty` (only FString `UStrProperty`), and Android has one output device anyway. **Threading:** UT99's Core has no threading API (the UE1 fork's `UnThread.h` — `appThreadSpawn`/`FMutex`/`FScopedLock`), so the driver's music-streaming thread uses a self-contained POSIX shim `Source/NOpenALDrv/NOpenALThread.h` that re-exports that exact API over pthreads (recursive mutex, since `UnregisterMusic`→`StopMusic` re-locks). **Gotcha:** that shim includes `<pthread.h>` in an already-Core-macro'd context, and Core's `UnFile.h` defines a function-like `clock(Timer)` timing macro that mangles `<time.h>`'s `clock_t clock(void)` — `#pragma push_macro("clock")`/`#undef clock`/`pop_macro` around the pthread include fixes it (the general "engine macro vs system header" trap; expect it for any system header pulled in after Core).

**The config `.ini`.** UT99 needs config that selects the Android drivers: `ViewportManager=NSDLDrv.NSDLClient`, a `[NSDLDrv.NSDLClient]` section, `GameRenderDevice`/`WindowedRenderDevice`/`RenderDevice=NOpenGLESDrv.NOpenGLESRenderDevice` (audio is force-set to `NOpenALDrv` in `InitAudio` regardless of the ini, so its `AudioDevice`/`UseSound` lines don't matter). A retail install's shipped ini points at Windows drivers and won't boot. This is now handled the same way UE1's is — see "Config ini redirection" below. `-GamePath=` (SAF/secondary-storage asset lookup) is also ported — see "SAF asset lookup (`-GamePath=`)". `-ResX=`/`-ResY=` was done earlier (Phase 2 resolution). The user-file (save/cache/screenshot) redirection to `user_files/` is also ported — see "User-file redirection" below.

- **Config ini redirection — both ini files live in `user_files/ut99/`.** UT99 splits config into **two** files (unlike Unreal's single `Unreal.ini`): `UnrealTournament.ini` (system/engine config, incl. the Android driver selection) and `User.ini` (keybinds/player setup). `appInit` (`Core/Src/UnMisc.cpp`) supports `-INI=`/`-USERINI=` natively and — unlike v200 — uses the parsed value **directly** (no `appBaseDir()` prefix), so the UE1 patch for absolute `-INI=` paths isn't needed here. `UT99Launcher` seeds both from known-good `assets/UnrealTournament.ini` + `assets/User.ini` into `user_files/ut99/` on first run (`checkForDownloads`, via `Utils.copyAsset`, only if absent so player changes persist) and always passes `-INI=<abs>` + `-USERINI=<abs>` in `getArgs()`. This keeps config on real, writable app storage regardless of whether the game data is on primary or SAF/secondary storage.

- **SAF asset lookup (`-GamePath=`), ported from `:Unreal`.** UT99's `appFindPackageFile` (`Core/Src/UnMisc.cpp` — v400's copy is here, not `UnPlat.cpp` as in v200, and `GSys->Paths` is now a `TArray<FString>` rather than v200's `char[96][16]`) builds each candidate path from `GSys->Paths(i)` (e.g. `..\Maps\*.unr`) as a CWD-relative string. SAFFAL's interceptor matches SAF-backed storage by literal string prefix and never resolves `../`, so relative paths silently fail on secondary/SD storage (same root cause as UE1 — see the UE1 "SAF … needs absolute paths" note). Fixed with the same `-GamePath=<abs>` argument, parsed lazily via `AndroidGamePath()` and applied by `AndroidAbsolutePath()`, which rewrites a leading `../` to `<GamePath>/…`. **Cleaner than the UE1 version:** v400's loop lets the rewrite run *before* the `"*"`-extension split (right after `appStrcat(Temp, *GSys->Paths(i))`), so there's no dangling-`Ext`-pointer problem and no need to copy the extension out first — done in both the fast-path and case-insensitive loops. `UT99Launcher.getArgs()` passes `-GamePath=` only when `subGame.rootPath == secondaryDirectory` (same condition UE1 uses); primary storage keeps the original relative/chdir behaviour. Not applied to `GSys->SavePath`/`CachePath` here — that's the separate user-file redirection patch below.

- **User-file redirection (saves/quicksaves/ladder progress, download cache, screenshots), ported from `:Unreal`.** `GSys->SavePath`/`CachePath` are plain ini-driven `UStrProperty`s (`SavePath=../Save`, `CachePath=../Cache` in `UnrealTournament.ini`) — CWD-relative like the asset-lookup paths above, so on SAF/secondary storage they'd silently fail the same way, and even on primary storage they wrote into the game's own install folder rather than the app's always-writable area. Fixed by an unconditional (not just SAF-gated) rewrite in `Core/Src/UnMisc.cpp`: `AndroidFixupGSysPathsOnce()` sets `GSys->SavePath`/`CachePath` to `<USER_FILES>/ut99/Save` and `.../Cache`, called once from `appFindPackageFile()` (mirrors `:Unreal`'s `AndroidFixupGSysPathsOnce()` in `Core/Src/UnPlat.cpp`). **v400 delta vs the v200 UE1 driver:** `SavePath`/`CachePath` are `FString`, not v200's fixed `char[96]`, so the fixup assigns (`FString::Printf`) rather than `snprintf`-ing into a fixed buffer — every other direct use (`UnGame.cpp` save-game code, `UnChan.cpp` download cache) already builds paths via `PATH_SEPARATOR`/`*GSys->SavePath`, so they pick up the redirect for free. Screenshots needed a separate fix, same as UE1: the `SHOT` console command (`Engine/Src/UnCamera.cpp`) built a bare relative `"ShotNNNN.bmp"` filename with no path at all, writing straight into CWD — patched (`__ANDROID__`-gated) to build a `<USER_FILES>/ut99/Screenshots` path and `GFileManager->MakeDirectory()` it (tree-create, since `Screenshots` doesn't already exist like `Save`/`Cache`'s parent does) before the exists-check loop. `user_files/ut99/` itself is created by `UT99Launcher.checkForDownloads()`'s ini-seeding (`Utils.copyAsset`'s `File.mkdirs()`) before the engine ever runs, so `Save`/`Cache`'s own non-recursive `MakeDirectory()` calls elsewhere in the engine always have an existing parent.

### Storage

Uses the OpenTouch storage model: legacy path `/OpenTouch/Sigma/` plus Android R+ scoped storage.

## Conventions & Gotchas

- **Comments: keep them simple and concise.** Prefer a single short line explaining the *why*.
- **ViewBinding is enabled** (`viewBinding true`).
- **Glide is pinned to 4.16.0** and `constraintlayout` to `1.1.3` / `recyclerview` to `1.3.2` for Nvidia Shield TV + API 19 compatibility — do not upgrade without testing on those targets.
- Native `.so` outputs in `src/main/libs/` are committed-elsewhere build artifacts; regenerate via `ndk-build` if a native submodule changes.
- The Gradle configuration cache is enabled.
